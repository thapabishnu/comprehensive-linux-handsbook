Environment, PATH, and startup files

"It works in my shell but not in cron" — the answer is here.

§1Why this matters

Every shell session inherits an environment and runs startup files. Once you understand which file gets read when, the mystery disappears.

§2Mental model

The shell has two big knobs:

  • Environment variables — name/value pairs exported to child processes (e.g. PATH, HOME, LANG). View with env. Set with export NAME=value.
  • Shell variables — set in the current shell only, not exported.

A subshell inherits env vars but cannot change its parent's. That's why cd in a script doesn't change your interactive directory.

Startup file order

Bash reads different files depending on how it was started:

Shell typeFiles read (in order)
Login shell (SSH, first console)/etc/profile → /etc/profile.d/*.sh → first found: ~/.bash_profile, ~/.bash_login, ~/.profile
Interactive non-login (new terminal in GUI)/etc/bash.bashrc → ~/.bashrc
Non-interactive (script run by bash script.sh)only $BASH_ENV if set

This is why a PATH export in ~/.bashrc works in your terminal but not under cron — cron's bash is non-interactive.

Practical pattern:

  • Put exports in ~/.profile or ~/.bash_profile.
  • Put aliases and functions in ~/.bashrc.
  • Make ~/.bash_profile source ~/.bashrc:
# ~/.bash_profile
[ -r ~/.bashrc ] && . ~/.bashrc

§4Guided walkthrough

# inspect
$ env | head -10
$ echo "$PATH"
$ echo "$HOME"
$ type cat
$ type ll        # probably "not found"
$ which ls
$ command -v ls

# add a personal bin dir
$ mkdir -p ~/bin
$ echo 'export PATH="$HOME/bin:$PATH"' >> ~/.profile
$ source ~/.profile

$ cat > ~/bin/mytool <<'EOF'
#!/bin/bash
echo "mytool says: $@"
EOF
$ chmod +x ~/bin/mytool
$ hash -r
$ mytool hi

# aliases
$ echo "alias ll='ls -la --color=auto'" >> ~/.bashrc
$ echo "alias ..='cd ..'" >> ~/.bashrc
$ . ~/.bashrc
$ ll
$ type ll        # "ll is aliased to ..."

# differentiate login vs non-login
$ bash -lc 'echo PATH=$PATH'
$ bash -c  'echo PATH=$PATH'
Mentor note · keep startup files quiet

Never put cd or anything that prints into ~/.profile. SSH-key login can fail silently if a startup file prints to stdout. Keep them quiet.

§5Gotchas

Gotcha · cron's environment is minimal

PATH is just /usr/bin:/bin. If your script uses tools in /usr/local/bin or ~/bin, it'll fail with "command not found." Set PATH in your script or use absolute paths.

Gotcha · ~/.bashrc vs ~/.bash_profile

One runs in interactive non-login, the other in login. Have one source the other to avoid double-config.

Mentor note · source vs bash script

source (or .) runs in the current shell. bash script.sh runs in a new shell. Variables set inside a script don't survive — unless you source.

Gotcha · don't put ./ in PATH

Running random commands from the current directory is a classic privilege-escalation vector.

Gotcha · alias doesn't take arguments

Use a function. alias gco='git checkout' works; alias mkcd='mkdir $1 && cd $1' does NOT — $1 is literal.

§6On-the-spot exercises

E15.1Find the full path of python3.

Show answer
$ which python3
$ command -v python3

E15.2Print PATH with each directory on its own line.

Show answer
$ echo "$PATH" | tr ':' '\n'

E15.3Add ~/scripts to PATH permanently for login shells.

Show answer
$ echo 'export PATH="$HOME/scripts:$PATH"' >> ~/.profile
$ source ~/.profile

E15.4Make vim your default editor.

Show answer
$ echo 'export EDITOR=vim' >> ~/.profile
$ echo 'export VISUAL=vim' >> ~/.profile

Two vars because some tools check one or the other. Git, crontab, sudoedit all honor $EDITOR.

E15.5Set an alias g=git permanently.

Show answer
$ echo "alias g='git'" >> ~/.bashrc
$ . ~/.bashrc

E15.6Write a function mkcd that creates a directory and cds into it.

Show answer
mkcd() { mkdir -p "$1" && cd "$1"; }
# put in ~/.bashrc to make it permanent

Functions take arguments; aliases don't.

E15.7Tell the shell to forget where it found ls.

Show answer
$ hash -r

E15.8Customize the prompt to show user@host and current dir.

Show answer
export PS1='[\u@\h \W]\$ '

\u = username, \h = short hostname, \W = basename of cwd. man bash + /PROMPTING for the full vocabulary.

§7Real-world sysadmin scenario

"My cron job worked yesterday, fails today."

*/5 * * * * /home/you/backup.sh

The script uses mytool from ~/bin. Cron doesn't read ~/.profile, so PATH is /usr/bin:/bin. Fix: absolute path, or set PATH inside the script:

#!/bin/bash
set -euo pipefail
export PATH="$HOME/bin:/usr/local/bin:/usr/bin:/bin"
mytool --backup

Or in the crontab itself:

PATH=/home/you/bin:/usr/local/bin:/usr/bin:/bin
*/5 * * * * /home/you/backup.sh

This one class of bug accounts for a huge fraction of "cron failures."

§8What to remember

  • PATH is colon-separated; first match wins.
  • Login shells read ~/.profile; interactive shells read ~/.bashrc.
  • Cron is non-interactive — set PATH explicitly in scripts.
  • Aliases for shortcuts; functions when you need arguments.
  • type cmd tells you exactly what the shell will run.
  • Keep startup files quiet.
