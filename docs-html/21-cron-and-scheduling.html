cron and scheduling

Backups, log rotation, cert renewal, monitoring scripts, daily reports — most server work runs on a schedule.

§1Why this matters

The two tools to know are cron (40 years of "good enough") and systemd timers (modern, more capable).

§2Mental model

A cron job has two parts: a schedule (5 time fields) and a command. Each user has their own crontab.

   ┌───────────── minute (0–59)
   │ ┌─────────── hour (0–23)
   │ │ ┌───────── day of month (1–31)
   │ │ │ ┌─────── month (1–12)
   │ │ │ │ ┌───── day of week (0–6, Sun=0)
   │ │ │ │ │
   * * * * *  command-to-run

Each field can be a number, *, a list (1,15,30), a range (9-17), or a step (*/15).

   */15 * * * *           every 15 minutes
   0 2 * * *              02:00 every day
   0 9 * * 1-5            09:00 weekdays
   30 4 1 * *             04:30 on the 1st of each month
   0 0 * * 0              midnight Sunday

§4Guided walkthrough

$ cat > ~/bin/log-uptime.sh <<'EOF'
#!/bin/bash
echo "$(date -Iseconds)  uptime: $(uptime -p)" >> "$HOME/uptime.log"
EOF
$ chmod +x ~/bin/log-uptime.sh

$ crontab -e
# add:  */5 * * * * /home/yourname/bin/log-uptime.sh

$ crontab -l
# wait 5 minutes...
$ tail -5 ~/uptime.log

at for one-shot

$ echo "shutdown -r" | at now + 5 minutes     # reboot in 5 (don't actually run!)
$ atq
$ atrm 1

§5systemd timers

/etc/systemd/system/mybackup.service:

[Unit]
Description=My backup

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh

/etc/systemd/system/mybackup.timer:

[Unit]
Description=Daily 02:00 backup

[Timer]
OnCalendar=*-*-* 02:00:00
Persistent=true

[Install]
WantedBy=timers.target
$ sudo systemctl daemon-reload
$ sudo systemctl enable --now mybackup.timer
$ systemctl list-timers --all | grep mybackup
$ journalctl -u mybackup.service

Persistent=true means if the machine was off at 02:00, the timer runs the next time it boots — cron can't do that without help.

§6Gotchas

Gotcha · cron's env is minimal

No PATH beyond /usr/bin:/bin, no aliases. Set PATH in the crontab or the script.

Mentor note · redirect output

Without MAILTO=, cron tries to mail you stdout/stderr. Best: redirect inside the entry: >> /var/log/myjob.log 2>&1.

Gotcha · cron uses /bin/sh, not bash

Your bash-isms may fail. Wrap in bash -c '...' or call a script with #!/bin/bash.

Gotcha · crontab -r wipes silently

Always crontab -l > /tmp/crontab-backup.txt first.

Mentor note · systemd timer failures

Timers don't fire if the unit is failed. systemctl reset-failed mybackup.service after fixing a bug.

§7On-the-spot exercises

E21.1Cron schedule for "every weekday at 5 PM."

Show answer
0 17 * * 1-5

E21.2"Every 10 minutes."

Show answer
*/10 * * * *

E21.3Cron entry running /usr/local/bin/backup.sh at 2 AM daily, logging to /var/log/backup.log.

Show answer
0 2 * * * /usr/local/bin/backup.sh >> /var/log/backup.log 2>&1

Critical: redirect both streams or you'll lose stderr.

E21.4Show your current crontab.

Show answer
$ crontab -l

E21.5Back up your crontab, then restore from it.

Show answer
$ crontab -l > /tmp/crontab.txt
$ crontab /tmp/crontab.txt

E21.6Schedule "reboot at 3 AM tomorrow" as a one-shot.

Show answer
$ echo "/sbin/reboot" | sudo at 3am tomorrow

E21.7Find when logrotate.timer fires next.

Show answer
$ systemctl list-timers --all | grep logrotate

E21.8A cron job runs but does nothing visible. Where to check?

Show answer
$ sudo grep CRON /var/log/syslog | tail -20
$ tail /tmp/myjob.log
$ env -i PATH=/usr/bin:/bin /path/to/myjob.sh

env -i clears the environment — reproduces "works in my shell, fails in cron."

§8Real-world sysadmin scenario

"The nightly backup didn't run last night."

# 1. Did cron fire?
$ sudo grep -E 'CRON.*(backup|root)' /var/log/syslog | tail -20

# 2. If yes, what did it output?
$ tail -50 /var/log/backup.log

# 3. Reproduce with cron's environment
$ sudo env -i PATH=/usr/bin:/bin /usr/local/bin/backup.sh

# 4. Look for resource starvation
$ journalctl --since "yesterday 02:00" --until "yesterday 02:30"

Two leading reasons cron jobs fail: minimal PATH; typo or stale comment.

Modern pattern: systemd timer with OnFailure=alert@%n.service for notifications.

§9What to remember

  • 5 fields: min hour dom mon dow. Use */N for "every N."
  • Cron's env is minimal. Set PATH and redirect output.
  • crontab -l > backup.txt before any risky edit.
  • For anything serious in 2026: systemd timer > cron.
  • Both still common. Read both.
