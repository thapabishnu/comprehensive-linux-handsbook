Processes and jobs

A box "running slow" or a service "stuck" almost always has its answer in ps, top, or kill.

§1Why this matters

Every running thing on Linux — your shell, the web server, your editor, the kernel modules pretending to be files — is or relates to a process. Inspecting, signaling, backgrounding, and (carefully) killing them is the bread and butter of operations.

§2Mental model

A process has:

  • A PID (process ID — a unique integer).
  • A PPID (parent process ID — the process that spawned it).
  • A UID/GID (whose privileges it runs with).
  • A state — Running, Sleeping, Stopped, Zombie.
  • Resources — open file descriptors, CPU/memory usage.

Every process has a parent. Trace up far enough and you reach PID 1 — init or systemd, the first thing the kernel runs.

   systemd (PID 1)
     ├─ sshd
     │   └─ sshd: yourname@pts/0
     │       └─ bash (your shell)
     │           └─ vim ← this is YOU right now
     ├─ nginx
     │   ├─ nginx: worker 1
     │   └─ nginx: worker 2
     └─ cron

You can ask the kernel anything about any process via /proc/<PID>/ — a virtual filesystem of process internals.

Foreground, background, jobs

When you run a command, by default it runs in the foreground — it owns your terminal until it exits. You can:

  • Add & at the end → runs in the background.
  • Press Ctrl+Z → suspends the foreground process (stops it; doesn't kill).
  • bg → resume a suspended process in the background.
  • fg → bring a backgrounded job back to the foreground.

A job is a shell concept (one or more processes started by one command line). Jobs have job IDs like %1, %2. Processes have PIDs.

§3Core commands

Inspecting

CommandWhat it shows
psYour processes in this terminal.
ps -efEvery process, full format (POSIX).
ps auxEvery process, BSD style (common on Linux).
ps -fU yournameEvery process owned by you.
ps -e --forestTree view of process parent/child.

pidof programPID(s) of running program.

htopFriendlier top.
pstreeProcess tree.

Signaling and killing

CommandEffect
kill PIDSend TERM — please exit.
kill -9 PIDSend KILL — immediate, unblockable. Last resort.
kill -HUP PIDReload config (many daemons honor this).
pkill patternKill by name pattern.
killall nameKill all processes with that exact name.

Common signals

Signal#Meaning
TERM15"Please terminate gracefully." Default.
KILL9"Die immediately." Can't be caught.
HUP1"Hang up." Often = "reload config."
INT2What Ctrl+C sends.
STOP19Suspend (can't be caught).
CONT18Continue a stopped process.
USR1, USR210, 12User-defined; apps choose.

Job control

CommandEffect
cmd &Start in background.
Ctrl+ZSuspend current foreground job.
jobsList your jobs.
fg %1Bring job 1 to foreground.
bg %1Continue job 1 in background.
kill %1Kill job 1 (note: %1 not PID).
disown %1Detach a job from this shell.
nohup CMD &Run a command immune to hangup.

For sessions that must survive — long deploys, training jobs — use tmux or screen.

§4Guided walkthrough

# inspect
$ ps
$ ps -ef | head -10
$ ps aux | wc -l
$ top                 # live; q to quit

# job control
$ sleep 60 &          # [1] 12345
$ jobs                # [1]+ Running   sleep 60
$ fg %1               # bring back; Ctrl+C to kill
$ sleep 30
# [Ctrl+Z] suspends — "[1]+ Stopped"
$ bg %1               # resume in background
$ jobs

# find and signal
$ sleep 999 &
$ pgrep sleep
$ kill $(pgrep sleep) # graceful TERM
$ pgrep sleep         # empty

nohup vs disown vs tmux

  • nohup CMD & — runs CMD, redirects output to nohup.out, immune to HUP. Good for fire-and-forget.
  • disown — used after the fact: cmd &; disown %1 removes from the shell's job table.
  • tmux — multiplexed terminal that runs on the server even if you disconnect. Reattach with tmux attach. Standard for any work over 30 seconds.

§5Gotchas

Gotcha · kill -9 doesn't run cleanup

No saved data, no released locks. Try kill (TERM) first, give it a few seconds, then escalate.

Gotcha · pkill -f matches too much

pkill nginx is safer than pkill -f api (which matches anywhere in the command line).

Tip · the grep-trap

ps aux | grep nginx matches the grep nginx process itself. Use pgrep, or ps aux | grep '[n]ginx'.

Mentor note · zombies aren't running

A zombie has exited but its parent hasn't read its exit code. They take no CPU. They do take a PID slot. Fix the parent, not the zombie.

Mentor habit · before kill -9

Ask: "what's the cost if this app can't run its cleanup handler?" For nginx, low. For a database, high. For an in-progress git push, possibly catastrophic.

§6On-the-spot exercises

E11.1Show every process owned by your user.

Show answer
$ ps -fU "$USER"

E11.2How many processes are currently running on this system?

Show answer
$ ps -e --no-headers | wc -l

E11.3Find the PID of sshd.

Show answer
$ pgrep -a sshd      # shows PIDs and the command line
$ pidof sshd

E11.4Start sleep 100 in the background. Confirm with jobs. Bring it to the foreground and kill it with Ctrl+C.

Show answer
$ sleep 100 &
$ jobs
$ fg %1               # Ctrl+C to terminate

E11.5In top, sort by memory instead of CPU.

Show answer

Inside top: press M. Press P to return to CPU sort. q to quit.

htop uses F6 to choose the sort field.

E11.6A misbehaving script is at PID 4242. Try graceful termination; force-kill only if that fails.

Show answer
$ kill 4242          # default TERM
$ ps -p 4242         # if still there:
$ kill -9 4242       # force kill

E11.7Display a process tree.

Show answer
$ pstree -p
$ ps -e --forest

E11.8Run tail -f /var/log/syslog so it survives you logging out.

Show answer
$ nohup tail -f /var/log/syslog > /tmp/syslog-tail.log 2>&1 &
$ disown

Or use tmux:

$ tmux new -s logwatch
$ tail -f /var/log/syslog
# Ctrl-b d  → detach (session keeps running)
# tmux attach -t logwatch  → reattach later

tmux is the better pattern for anything you'll want to look at again.

§7Real-world sysadmin scenario

Box is at 100% CPU; no obvious culprit. You SSH in:

$ uptime
$ top -bn1 | head -20

top shows mysqld at 380% CPU. That needs a deeper look:

$ pgrep -a mysqld
$ ps -o pid,user,%cpu,%mem,start,etime,cmd -p 12345
$ ls -l /proc/12345/cwd
$ cat /proc/12345/status | head -20

Evidence before action. A bad query is hammering MySQL — fix is in the DB, not the kill. kill -9 would lose in-flight transactions and watch the problem return on the next query.

For stuck services: systemctl restart mysql goes through the unit's shutdown sequence. We meet systemd in module 17.

§8What to remember

  • Every process has a PID and a parent. pstree and ps --forest show the family tree.
  • Default kill is TERM (graceful). -9 KILL is last resort.
  • Ctrl+Z suspends, fg/bg resumes, & backgrounds.
  • For long jobs that must survive your shell: nohup or tmux.
  • top / htop first; ps for snapshots; pgrep/pkill for name matches.
  • Zombies are dead; fixing the parent is the fix.
