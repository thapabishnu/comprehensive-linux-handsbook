grep and regular expressions

Regex is the most reusable skill in this entire course. Five hours of it pays back for a career.

§1Why this matters

Every Linux job you ever do involves "find the lines that match X." Configs, logs, source code, output of other commands. grep is how. And grep's pattern language — regular expressions — carries into Python, JavaScript, every editor, every database, every cloud console.

§2Mental model

grep reads input line-by-line and prints lines matching a pattern. The pattern is either:

  • A fixed string — grep error log.txt (treats error literally)
  • A regular expression — grep '^ERROR' log.txt (treats ^ as "start of line")

There are several flavors of regex:

FlavorTool flagNotes
Basic Regex (BRE)default grep?, +, {} need backslashes: \?, \+
Extended Regex (ERE)grep -E (or egrep)Modern operators work without escaping
Perl-compat (PCRE)grep -PMost powerful; lookaheads, lazy quantifiers
Fixed stringgrep -F (or fgrep)No regex, fastest, safest when you mean a literal

For most daily work, ERE (grep -E) is the sweet spot.

Mentor note · anchored vs floating

A pattern is anchored (matches at a position) or floating (matches anywhere). ^abc is anchored to the start of a line; abc floats. Knowing which you want is half of writing a correct regex.

§4Regex building blocks (ERE)

Single-character matchers

PatternMatches
aThe literal character a
.Any single character
[abc]One of a, b, c
[a-z]One lowercase letter
[A-Z0-9]One uppercase letter or digit
[^abc]One character not in the set
\dDigit (PCRE only — for grep, use [0-9])
\sWhitespace (PCRE only — use [[:space:]])

Anchors

PatternMatches
^Start of line
$End of line
\bWord boundary (in many flavors; safer: use -w)

Quantifiers (how many of the previous thing)

PatternMeaning
*Zero or more
+One or more (ERE)
?Zero or one (ERE)
{n}Exactly n
{n,}n or more
{n,m}Between n and m

Grouping and alternation

PatternMeaning
(abc)Group — treat as one unit
a|ba or b (alternation in ERE)
(cat|dog)"cat" or "dog"

POSIX character classes (portable)

ClassEquivalent
[[:digit:]][0-9]
[[:alpha:]][A-Za-z]
[[:alnum:]][A-Za-z0-9]
[[:space:]]space, tab, newline
[[:upper:]][A-Z]
[[:lower:]][a-z]
Mentor note · use -E by default

In BRE, +, ?, {}, |, () are literal characters. To use them as operators you must backslash. ERE makes them operators directly. Use grep -E unless you have a reason not to.

§5Guided walkthrough

$ cd ~/linux-sandbox/text/

# fixed string
$ grep 200 access.log              # any line containing "200"
$ grep ' 200 ' access.log          # tighter — must have spaces around

# inverted
$ grep -v 200 access.log

# case-insensitive
$ grep -i error lorem.txt

# anchors
$ grep '^192' access.log
$ grep '^[0-9]' access.log
$ grep -E '^[A-Z][a-z]+' names.txt

# alternation
$ grep -E '401|403|404' access.log

# quantifier — extended
$ grep -E ' (4|5)[0-9]{2} ' access.log

# whole word
$ grep -w book ~/linux-sandbox/text/lorem.txt

# count + recursive
$ grep -rc TODO ~/linux-sandbox/

# context
$ grep -n -A 1 -B 1 401 access.log

# only the match
$ grep -oE '[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+' access.log | sort -u

That last line is genuinely useful — -o prints only the matched piece. Combined with regex it's a featherweight extractor.

§6Gotchas

Gotcha · always quote your patterns

grep ^ERROR file lets the shell try to interpret ^. grep '^ERROR' file doesn't. Single quotes also protect from $, *, etc.

Gotcha · BRE vs ERE

grep '(cat|dog)' file in BRE searches for the literal string (cat|dog). Use -E or escape: \(\) and \|.

Mentor note · greedy by default

Standard regex .* is greedy — matches as much as possible. There's no lazy ? in BRE/ERE; you need -P for that. Workaround: [^"]* (between quotes, don't cross another quote).

Gotcha · grep matches anywhere on the line

Use ^...$ to anchor to the whole line, or -x to match only complete lines.

Tip · grep -F is faster

If you're searching for a literal IP address, grep -F '10.0.0.55' skips the regex engine.

Tip · don't grep yourself

ps aux | grep nginx matches the literal grep nginx process. Use ps aux | grep '[n]ginx' — the bracket trick makes the regex not match itself.

§7On-the-spot exercises

E9.1In names.txt, find every last name beginning with R.

Show answer
$ grep '^R' ~/linux-sandbox/text/names.txt

^ = start of line.

E9.2Count how many words in the dictionary contain the substring book.

Show answer
$ grep -c book /usr/share/dict/words

E9.3Count words that start with book.

Show answer
$ grep -c '^book' /usr/share/dict/words

E9.4From access.log, show all 4xx and 5xx responses.

Show answer
$ grep -E ' (4|5)[0-9]{2} ' ~/linux-sandbox/text/access.log

Leading/trailing spaces avoid matching "401" inside a longer number.

E9.5Display only active config lines in /etc/ssh/ssh_config (skip blank lines and comments).

Show answer
$ grep -Ev '^\s*#|^\s*$' /etc/ssh/ssh_config

Two patterns OR'd, inverted = "only real config lines."

E9.6Recursively, in /etc/, find every file mentioning your username.

Show answer
$ sudo grep -r -l "$USER" /etc/ 2>/dev/null

E9.7Show line numbers of every export line in ~/.bashrc.

Show answer
$ grep -nE '^[[:space:]]*export ' ~/.bashrc

E9.8Extract just the unique IP addresses from access.log.

Show answer
$ grep -oE '^[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+' ~/linux-sandbox/text/access.log | sort -u

-o prints only the matched piece. sort -u deduplicates.

E9.9In lorem.txt, find lines containing both regex and pattern.

Show answer
$ grep regex ~/linux-sandbox/text/lorem.txt | grep pattern

No AND in regex — chain greps via a pipe.

E9.10Find every line of /etc/passwd whose login shell is /bin/false or /usr/sbin/nologin.

Show answer
$ grep -E ':(/bin/false|/usr/sbin/nologin)$' /etc/passwd

§8Real-world sysadmin scenario

Incident. A customer reports 500 errors. You need to know: how many, which endpoints, which IPs.

$ sudo zgrep ' 500 ' /var/log/nginx/access.log* | wc -l

zgrep works on .gz rotated logs without un-gzipping. Total: 412.

Now break it down:

$ sudo zgrep ' 500 ' /var/log/nginx/access.log* \
  | grep -oE '"[A-Z]+ [^"]+"' \
  | sort | uniq -c | sort -rn | head -10

In 30 seconds, you have the top 10 endpoints responsible for the 500s. grep + sort + uniq -c + sort -rn + head is the most useful five-stage incantation in this entire guide.

§9What to remember

  • grep reads input line-by-line, prints matches.
  • Use grep -E (ERE) unless there's a reason not to.
  • Anchors: ^ start, $ end. Char classes: [abc], [^abc], [a-z].
  • Quantifiers: *, +, ?, {n}, {n,m}.
  • Useful flags: -i, -v, -n, -c, -l, -r, -w, -o, -A/-B/-C.
  • Quote your patterns. Always.
  • grep ... | sort | uniq -c | sort -rn — the investigation pipeline.
