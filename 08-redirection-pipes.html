Redirection and pipes

Unix's superpower isn't any one tool — it's that small tools compose. Redirection and piping are the wiring.

§1Why this matters

Once you internalize the three standard streams and the operators that bend them, you stop seeing commands as isolated and start seeing pipelines.

§2Mental model

Every program has three open file descriptors by default:

FDNameDefault destination
0stdinYour keyboard
1stdoutYour terminal
2stderrYour terminal

You can redirect any of them:

    program  ──1──►  stdout       │  redirect with  >   >>
    program  ──2──►  stderr       │  redirect with  2>  2>>
    program  ◄──0──  stdin        │  redirect with  <

And you can wire one program's stdout into another's stdin — that's the pipe:

    cmd1  ──1──►  ──0──►  cmd2  ──1──►  ──0──►  cmd3
                  pipe              pipe

Two huge insights:

  1. stdout and stderr are separate. A command that fails quietly on the terminal is often complaining loudly on stderr — you just merged them. To see them apart: cmd > out 2> err.
  2. Pipes only connect stdout to stdin. If you want stderr in the pipeline too, merge it first: cmd 2>&1 | next.

§3Core operators

OperatorEffect
cmd > filestdout → file (overwrite)
cmd >> filestdout → file (append)
cmd 2> filestderr → file
cmd 2>> filestderr append
cmd > out 2> errsplit stdout and stderr
cmd > file 2>&1both to same file (note: order matters!)
cmd &> filebash shorthand for > file 2>&1
cmd 2>&1 | nextfeed both streams into the pipeline
cmd < fileread stdin from a file
cmd1 | cmd2stdout of cmd1 → stdin of cmd2
cmd | tee fileprint to terminal and save to file
cmd | tee -a filesame, but append
cmd > /dev/nulldiscard stdout
cmd 2> /dev/nulldiscard stderr
cmd >/dev/null 2>&1discard everything (the classic "silence" idiom)
Mentor note · why order matters

2>&1 says "send fd 2 to wherever fd 1 currently goes." That's why cmd > file 2>&1 works but cmd 2>&1 > file does not — in the second form, 2>&1 is set up before > moves fd 1, so stderr still ends up on the terminal.

§4Guided walkthrough

# stdout to a file
$ echo "hello" > greeting.txt
$ cat greeting.txt

# overwrite
$ echo "again" > greeting.txt
$ cat greeting.txt           # only "again"

# append
$ echo "and again" >> greeting.txt
$ cat greeting.txt

# stderr separately
$ ls /nonexistent 2> errors.log
$ cat errors.log

# merge both streams
$ ls /etc /nonexistent > combined.log 2>&1
$ cat combined.log

# discard noise
$ find / -name passwd 2>/dev/null | head -5

# piping
$ ls /etc | wc -l            # how many entries in /etc?
$ ls -lt /var/log | head -5  # 5 most recently modified logs

# tee — see and save
$ echo "deployed" | tee -a /tmp/deploy.log

# input redirection
$ wc -l < /etc/passwd        # wc reads from a redirected stdin

Try this drill: count how many of your man pages are in section 5:

$ man -k . 2>/dev/null | grep '(5)' | wc -l

Three-stage pipeline doing real work in a single line.

§5Gotchas

Gotcha · > truncates instantly

Even if the command fails. echo > important.txt empties the file before echo runs. Use >> if you mean append, and set -o noclobber in scripts.

Gotcha · order of > and 2>&1

Always put 2>&1 after > file. Otherwise stderr won't follow stdout into the file.

Gotcha · tee doesn't capture stderr

cmd | tee log only tees stdout. To tee both: cmd 2>&1 | tee log.

Mentor note · pipelines run in parallel

cmd1 | cmd2 doesn't wait for cmd1 to finish — both run, with the pipe as a streaming buffer. For ordered processing keep that in mind.

Mentor note · exit codes in pipelines

By default the pipeline's exit code is cmd2's. To check cmd1 failed: set -o pipefail makes the pipeline fail if any stage fails. Critical for scripts.

§6On-the-spot exercises

E8.1Save the output of ls /etc to a file called etc-listing.txt.

Show answer
$ ls /etc > etc-listing.txt

E8.2Append today's date to ~/work.log.

Show answer
$ date >> ~/work.log

>> (not >!) — otherwise you'd overwrite the log.

E8.3Run ls /etc /nope and put the listing in out.txt, the errors in err.txt.

Show answer
$ ls /etc /nope > out.txt 2> err.txt

E8.4Same command, but combine everything into all.log.

Show answer
$ ls /etc /nope > all.log 2>&1
# or bash shorthand:
$ ls /etc /nope &> all.log

E8.5Count how many .conf files are directly under /etc/.

Show answer
$ ls /etc/*.conf | wc -l

Globbing (mod 04) + a pipe + wc.

E8.6Show all 200 responses from the access.log sandbox, then count them.

Show answer
$ grep ' 200 ' ~/linux-sandbox/text/access.log
$ grep ' 200 ' ~/linux-sandbox/text/access.log | wc -l

Or chain into one display:

$ grep ' 200 ' ~/linux-sandbox/text/access.log | tee /tmp/200s.log | wc -l

tee lets you save the matches AND print the count.

E8.7Discard find's noisy "Permission denied" errors while searching /etc for files containing "host" in the name.

Show answer
$ find /etc -name '*host*' 2>/dev/null

/dev/null is the bit bucket — anything sent there is destroyed.

E8.8Without using cat, send the contents of ~/linux-sandbox/text/names.txt into wc -l.

Show answer
$ wc -l < ~/linux-sandbox/text/names.txt

< redirects the file in as stdin. wc would also accept a filename directly — but < is the redirection form, which is what the exercise asks for.

E8.9Build a one-liner: "the 3 most active IPs in the access log."

Show answer
$ awk '{print $1}' ~/linux-sandbox/text/access.log \
  | sort | uniq -c | sort -rn | head -3

5 stages: extract field, sort to group, count duplicates, sort numerically descending, take top 3. We'll meet awk properly in module 13.

§7Real-world sysadmin scenario

Long-running deploy script. A teammate writes a script that runs for 20 minutes and you want to know:

  1. Did it succeed?
  2. What did it print?
  3. Were there any errors that didn't make it to the visible output?
$ ./deploy.sh > /tmp/deploy.out 2> /tmp/deploy.err
$ echo "Exit: $?"

If $? is 0, success. The two files contain stdout and stderr separately. If you wanted to watch in real time AND keep a record:

$ ./deploy.sh 2>&1 | tee /tmp/deploy.log
$ echo "Pipeline exit: ${PIPESTATUS[0]}"

PIPESTATUS[0] gives the first stage's exit code. Without that, $? would only tell you that tee succeeded — which is almost always true. This is what separates "I ran the script" from "I ran the script and know whether it worked."

§8What to remember

  • Three streams: stdin (0), stdout (1), stderr (2). Memorize the numbers.
  • > overwrites, >> appends. 2> for stderr.
  • 2>&1 after the stdout redirect, never before.
  • | connects stdout → stdin. Use 2>&1 | to pipe both.
  • tee shows AND saves. /dev/null is the silence switch.
  • In scripts, set -o pipefail + ${PIPESTATUS[@]} are how you check pipeline failures.
