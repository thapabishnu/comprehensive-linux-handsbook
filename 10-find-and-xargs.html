find and xargs

Globs match files in one directory. find walks an entire tree and matches by any criterion — then acts on each match.

§1Why this matters

The combination of find and xargs is how sysadmins do bulk operations safely: "delete all the 30-day-old log files," "set every directory to mode 755 but every file to 644," "tar up everything modified yesterday."

§2Mental model

find has a slightly weird syntax that throws beginners. It's actually quite consistent:

   find  STARTING_PATH(s)   TESTS   ACTION
   ────  ────────────────   ─────   ──────
   find  /var/log           -name '*.gz' -mtime +7   -delete
  • Starting path — where to begin walking.
  • Tests — filters that all must match (implicit AND).
  • Action — what to do with each match. Default is -print.

find is exhaustive by default — it descends into every subdirectory. That power is why it's useful and why you must point it carefully.

§3Core find tests

<test> -o <test>Logical OR (default is implicit AND)
\( ... \)Group expressions (parens)

Actions

ActionWhat it does
-printPrint the path (default)
-print0NUL-separated (safe for filenames with spaces)
-lsLong-format listing like ls -l
-deleteDelete the match
-exec CMD {} \;Run CMD per match. {} is the path.
-exec CMD {} +Same but batches paths (faster)
-quitStop after the first match
Gotcha · -delete is final

Always run the same command with -print first to see what would be deleted. Then change -print to -delete.

§4xargs — the partner tool

xargs reads lines from stdin and turns them into arguments for a command.

$ find . -name '*.log' | xargs gzip
$ find . -name '*.log' -print0 | xargs -0 gzip   # NUL-safe
FlagEffect
-0Input separated by NUL (pair with find -print0)
-n NUp to N args per command call
-P NRun up to N copies in parallel
-I {}Use {} as placeholder
-rDon't run if no input

§5Guided walkthrough

$ cd ~/ncs205-sandbox/tree/

# the simplest find
$ find .                              # every path, recursively

# filter by name
$ find . -name '*.md'
$ find . -iname '*.MD'

# filter by type
$ find . -type f
$ find . -type d

# combine tests (AND is implicit)
$ find . -type f -name '*.log'

# size
$ find . -type f -size +1c

# negate
$ find . -not -name '*.md'

# OR — must group expressions
$ find . \( -name '*.log' -o -name '*.tmp' \) -type f

# show full listing
$ find . -type f -ls

# run a command per match — old style
$ find . -type f -exec wc -l {} \;

# run once with many args — faster
$ find . -type f -exec wc -l {} +

# safer alternative with xargs
$ find . -type f -print0 | xargs -0 wc -l

# age-based cleanup (DRY-RUN FIRST)
$ find ~/ncs205-sandbox -name '*.tmp' -mtime +0 -print
# find ~/ncs205-sandbox -name '*.tmp' -mtime +0 -delete

Note the print-then-delete pattern. Bake it into your habits.

§6Gotchas

Gotcha · quote your -name

find . -name *.log lets the shell expand *.log before find runs. Use -name '*.log'.

Mentor note · -mtime is days; -mmin is minutes

Leading + = more than, - = less than, no sign = exactly N. -mtime 0 matches files modified in the last 24 hours.

Gotcha · -exec ... \; is slow

It runs the command once per file. Use -exec ... + to batch — way faster.

Gotcha · spaces in filenames break naive xargs

find ... | xargs splits on whitespace, so "report v2.txt" becomes two args. Use find ... -print0 | xargs -0.

Mentor note · -perm nuances

-perm 644 means exactly 644. -perm -644 means at least those bits. -perm /644 means any of those bits.

Tip · stay on one filesystem with -xdev

find / can be slow because it traverses every mount. -xdev stops at filesystem boundaries.

§7On-the-spot exercises

Use the sandbox tree at ~/ncs205-sandbox/tree/.

E10.1List every .md file under the tree.

Show answer
$ find ~/ncs205-sandbox/tree -name '*.md'

E10.2List every directory under the tree.

Show answer
$ find ~/ncs205-sandbox/tree -type d

E10.3Find every file modified in the last 7 days under your home directory.

Show answer
$ find ~ -type f -mtime -7

-7 = "less than 7 days ago," +7 = "more than 7 days ago."

E10.4Find every empty file under /tmp/.

Show answer
$ find /tmp/ -type f -empty 2>/dev/null

E10.5Find every .log or .tmp file in /var/.

Show answer
$ sudo find /var/ \( -name '*.log' -o -name '*.tmp' \) -type f 2>/dev/null

Without the parens, -o binds looser than -type, and the meaning shifts.

E10.6Show the size and path of every file over 1 MB in /var/log/.

Show answer
$ sudo find /var/log/ -type f -size +1M -ls

E10.7Delete every *.tmp file in ~/ncs205-sandbox/tree/. First print what would be deleted, then delete.

Show answer
$ find ~/ncs205-sandbox/tree -name '*.tmp' -print     # confirm
$ find ~/ncs205-sandbox/tree -name '*.tmp' -delete    # actually delete

The print-then-delete habit. Internalize it.

E10.8Recursively chmod 644 every file (but not directory) under ~/ncs205-sandbox/.

Show answer
$ find ~/ncs205-sandbox -type f -exec chmod 644 {} +

E10.9Find every file in /etc/ owned by you.

Show answer
$ find /etc -user "$USER" 2>/dev/null

E10.10Count the total disk usage of all .log files under /var/log/ using find + xargs.

Show answer
$ sudo find /var/log -type f -name '*.log' -print0 \
  | xargs -0 du -ch \
  | tail -1

§8Real-world sysadmin scenario

The dreaded "disk filling up" — but smartly this time:

# Top 10 biggest files anywhere (skipping /proc, /sys)
$ sudo find / -xdev -type f -not -path '/proc/*' -not -path '/sys/*' \
    -printf '%s %p\n' 2>/dev/null \
  | sort -rn | head -10 \
  | awk '{ printf "%6.1fM  %s\n", $1/1024/1024, $2 }'

One-liner you copy into your runbook. Once you have the top 10, you decide which to act on — and find can take that action too:

# Compress all .log files in /var/log/myapp older than 7 days
$ sudo find /var/log/myapp -type f -name '*.log' -mtime +7 -exec gzip {} +

§9What to remember

  • find PATH TESTS ACTION — that's the whole pattern.
  • Quote -name patterns.
  • -print first, -delete second. Always.
  • -exec CMD {} + is the modern, fast form.
  • xargs -0 + find -print0 is the safe way to pipe filenames.
  • For one-filesystem searches, add -xdev.
