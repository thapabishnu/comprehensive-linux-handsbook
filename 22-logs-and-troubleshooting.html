Logs and troubleshooting

Every previous module has said "check the logs." Now we cover it properly. Reading logs well is the difference between sysadmin and senior sysadmin.

§1Why this matters

The commands are easy; the discipline of what to look at, in what order is the real skill.

§2Mental model

Linux logs live in two places:

  1. /var/log/ — text files written by syslog/rsyslog. Old, universal, easy to grep.
  2. systemd journal — structured binary log, queried via journalctl. Modern, richer.

Most modern distros use both. The right ordering when something breaks:

1. journalctl -u UNIT --since "X" -p err     ← what does the failing service say?
2. dmesg --since "X"                          ← anything from the kernel?
3. /var/log/syslog (or messages)              ← system-wide events
4. application-specific logs                  ← /var/log/nginx/, /var/log/mysql/

§5Guided walkthrough

$ sudo tail -50 /var/log/syslog
$ sudo tail -F /var/log/syslog
# in another terminal:
$ sudo systemctl restart ssh

# the journal
$ journalctl -e -n 30
$ journalctl -u ssh -n 10
$ journalctl --since "10 min ago"
$ journalctl -p err -b
$ journalctl -k --since "1 hour ago"

# kernel ring buffer
$ dmesg -T --level=err,warn | tail

# system snapshot
$ uptime
$ free -h
$ df -h
$ top -bn1 | head -20

# login history
$ last | head
$ sudo lastb | head
Mentor habit · five-command health sweep

When you don't know what's wrong, run uptime, free -h, df -h, top -bn1, journalctl -p err -b | tail -30. Always run this before you start randomly restarting things. It gives you 80% of what you need to form a hypothesis.

§6Common failure patterns

SymptomFirst place to look
Service won't startsystemctl status SVC then journalctl -u SVC -p err
Box "feels slow"uptime → top → iostat
"Disk full"df -h → du -sh /var/* /tmp/* /home/* → find / -size +1G
Can't log insudo tail /var/log/auth.log or journalctl -u ssh
Random rebootslast reboot, dmesg, journalctl --list-boots

§7Gotchas

Mentor note · time matters

Pin every incident to a precise time window. --since / --until saves enormous noise.

Gotcha · /var/log/syslog rotates

Look in syslog.1, syslog.2.gz for older history. zgrep reads .gz directly.

Gotcha · dmesg is a ring buffer

Oldest messages get evicted. journalctl -k keeps them longer.

Mentor note · journal persistence

Without /var/log/journal/ the journal is volatile. sudo mkdir -p /var/log/journal && sudo systemctl restart systemd-journald.

Tip · don't cat a huge log

less it, or page with tail/head.

Mentor maxim

A "fix" with no log evidence is a guess. If your change didn't appear in logs as the cause, your fix won't appear in logs as the cure.

§8On-the-spot exercises

E22.1Show last 50 lines of /var/log/syslog.

Show answer
$ sudo tail -n 50 /var/log/syslog

E22.2Follow the journal live, filtered to error priority.

Show answer
$ sudo journalctl -f -p err

E22.3Show all kernel messages from the last hour.

Show answer
$ sudo journalctl -k --since "1 hour ago"

E22.4Find every line in this boot's journal that mentions OOM.

Show answer
$ sudo journalctl -b -k | grep -i oom

"Out of memory" / "Killed process" = the kernel's OOM killer fired.

E22.5Failed SSH login attempts recently?

Show answer
$ sudo lastb | head
$ sudo grep "Failed password" /var/log/auth.log | wc -l

Internet-exposed SSH typically sees thousands per day. Why we set up keys + 2FA in module 19.

E22.6Every error log line from nginx in last 24h.

Show answer
$ sudo journalctl -u nginx -p err --since "24 hours ago"
$ sudo less +G /var/log/nginx/error.log

E22.7Show every log line tagged with PID 4242.

Show answer
$ journalctl _PID=4242

_PID= is a journal match. Same syntax for _UID=, _SYSTEMD_UNIT=.

E22.8Identify the 5 largest directories under /var/.

Show answer
$ sudo du -h /var/* 2>/dev/null | sort -rh | head -5

E22.9Tail-follow auth.log in real time.

Show answer
$ sudo tail -F /var/log/auth.log

E22.10Times of the last 5 reboots.

Show answer
$ last reboot | head -5
$ journalctl --list-boots

§9Real-world sysadmin scenario

3 AM page: "web-02 not responding."

# Health snapshot
$ uptime                         # load avg: 18.2 12.1 6.4  ← bad and getting worse
$ free -h                        # 7.8G used / 8.0G total, swap 100%
$ df -h                          # /var 99% full
$ journalctl -p err -b | tail -30
# "Could not allocate space for log entry"
# "Out of memory: Killed process 1234 (nginx)"

Full story: disk filled → nginx couldn't write → swap filled with bloated processes → OOM killer murdered things. The fix isn't "restart nginx" — it's "clear /var/log."

$ sudo du -sh /var/log/* | sort -rh | head -5
# /var/log/myapp/   12G  ← log rotation isn't running
$ sudo truncate -s 0 /var/log/myapp/big.log
$ sudo systemctl restart myapp
$ sudo logrotate -f /etc/logrotate.d/myapp

Total time: under 5 minutes. Real diagnosis followed by real fix. You did not restart things until you understood why they had failed. That's the principle that ran through this entire guide.

§10What to remember

  • Two log systems: /var/log/* and the journal.
  • Health snapshot first: uptime, free -h, df -h, top -bn1, journalctl -p err -b | tail.
  • Always pin to a time window.
  • Read logs before you act. Restart-without-understanding is the #1 way to make incidents worse.
