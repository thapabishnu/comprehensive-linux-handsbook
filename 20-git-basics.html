MODULE20 TIME~30 min

git for sysadmins

Every change you make to a config, every script you write, every dotfile you tweak — should be versioned.

§1Why this matters

When a config worked yesterday and doesn't today, git log and git diff tell you exactly what changed and why.

§2Mental model

A repository stores snapshots of a directory's state. Each snapshot is a commit with a SHA, a parent pointer, and a message.

A branch is a moveable pointer to a commit. HEAD points to whichever commit/branch you're currently "on."

   commit C ──► commit B ──► commit A (root)
       ▲
     main, HEAD

Three states your files can be in:

StateMeans
Working directoryFiles you can see and edit
Staging area (index)Changes marked for the next commit
CommittedSnapshot saved permanently

git add moves changes from working → staging. git commit moves staging → committed.

One-time setup

$ git config --global user.email "you@example.com"
$ git config --global init.defaultBranch main

§4Guided walkthrough

$ mkdir ~/configs && cd ~/configs
$ git init
$ echo "*.swp" > .gitignore
$ echo "alias ll='ls -la'" > bashrc-snippets.sh

$ git status
$ git add .gitignore bashrc-snippets.sh
$ git status
$ git commit -m "initial: gitignore + first bash alias"

# make a change
$ echo "alias ..='cd ..'" >> bashrc-snippets.sh
$ git diff
$ git add bashrc-snippets.sh
$ git commit -m "alias: add .. for cd .."

$ git log --oneline
$ git show HEAD
$ git blame bashrc-snippets.sh

Pushing to GitHub

$ git remote add origin git@github.com:yourname/configs.git
$ git push -u origin main
Mentor habit · small, focused commits

One concern per commit. git add -p lets you split a file's changes into multiple commits — invaluable for readable history.

§5Gotchas

Gotcha · don't commit secrets

Once pushed, they're public forever — history is preserved. Add to .gitignore before the first commit.

Mentor note · git pull can surprise

git pull = fetch + merge and can produce surprise merge commits. Some prefer git pull --rebase for linear history.

Gotcha · destructive commands

git reset --hard and git push --force can vanish coworkers' work. Use --force-with-lease and avoid force-pushing shared branches.

Mentor note · detached HEAD

Happens when you checkout a commit hash. Any commits there are "lost" unless you create a branch: git switch -c rescue-branch.

Tip · commit messages are documentation

"Fix bug" is useless. "Fix off-by-one in date parsing when timezone is DST-ambiguous" is gold a year later.

§6On-the-spot exercises

E20.1Initialize a git repo in ~/lab20 and commit a hello-world script.

Show answer
$ mkdir -p ~/lab20 && cd ~/lab20
$ git init
$ echo '#!/bin/bash' > hello.sh
$ echo 'echo hello' >> hello.sh
$ chmod +x hello.sh
$ git add hello.sh
$ git commit -m "add hello.sh"

E20.2Show what's in the latest commit.

Show answer
$ git show HEAD

HEAD~1 = one commit ago, etc.

E20.3Make a change. Use git diff to see it.

Show answer
$ echo 'date' >> hello.sh
$ git diff

E20.4Stage the change. Use git diff --cached to see it.

Show answer
$ git add hello.sh
$ git diff             # nothing
$ git diff --cached    # the staged change

E20.5Unstage and discard the change.

Show answer
$ git restore --staged hello.sh
$ git restore hello.sh

E20.6Show history of hello.sh with full diffs.

Show answer
$ git log -p hello.sh

E20.7Add a remote origin, then list remotes.

Show answer
$ git remote add origin git@github.com:you/configs.git
$ git remote -v

E20.8Untrack a file you committed by mistake (keep it in working dir).

Show answer
$ echo "passwords.txt" >> .gitignore
$ git rm --cached passwords.txt
$ git commit -m "untrack passwords.txt"

File is still in earlier history. For genuine secrets, rotate the secret (faster than rewriting history).

§7Real-world sysadmin scenario

Versioning /etc on a server. Turn /etc into a git repo:

$ cd /etc
$ sudo git init
$ sudo bash -c 'cat > .gitignore' <<'EOF'
shadow
shadow-
gshadow
gshadow-
private/
ssh/ssh_host_*_key
EOF

$ sudo git add .
$ sudo git commit -m "baseline: $(hostname) $(date -Iseconds)"

Whenever you edit a config:

$ sudo vi /etc/nginx/nginx.conf
$ sudo git -C /etc diff
$ sudo git -C /etc add nginx/nginx.conf
$ sudo git -C /etc commit -m "nginx: tune worker_connections to 4096"

Two months later, the junior asks "why is worker_connections 4096?" You: sudo git -C /etc log -p nginx/nginx.conf — full history with reasons.

Real sysadmins use etckeeper, a tool that does this with extras (auto-commits on package install).

§8What to remember

  • Repos store snapshots = commits. Branches are pointers.
  • Working dir → staging → committed. add then commit.
  • Write good commit messages.
  • git status and git diff are your two most-used commands.
  • git log -p file for the why; git blame file for the who.
  • Never commit secrets. .gitignore before git add.
