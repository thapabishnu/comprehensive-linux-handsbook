MODULE18 TIME~30 min

Networking basics

Debug bottom-up: link → IP → port → protocol. Each layer has a tool.

§1Why this matters

Production Linux is a networking platform. When "it works on localhost but not from outside," when DNS misbehaves, when a port shows open from one machine but closed from another — these tools are the difference between guessing and knowing.

§2Mental model

Networking has layers; for sysadmin work, four matter:

   L7  HTTP / SSH / SMTP / DNS              ← what apps speak
   L4  TCP / UDP / ports (16-bit numbers)   ← connections
   L3  IP addresses, routes                 ← addressing
   L2  MAC addresses, the local link        ← Ethernet/Wi-Fi

When something doesn't work, go bottom-up: is the cable up (L2), do I have an IP (L3), is the port open (L4), is the protocol responding (L7)?

§3Core commands

Layer 2 / 3 — links, addresses, routes

CommandWhat it shows

Layer 4 — TCP/UDP, ports

CommandWhat it shows
ss -tulpnListening sockets (TCP/UDP, with PID, port, program)
ss -tanAll TCP connections (numeric)
nc -zv host portTest if a remote port is open
nc -lv -p 12345Open a listener for testing
ufw status / iptables -LShow firewall rules

netstat is the old version of ss — slower, sometimes missing. Use ss.

§4Guided walkthrough

# Layer 2/3 — what links and addresses do we have?
$ ip link
$ ip -4 addr
$ ip route
$ ip route get 8.8.8.8

# Reachability
$ ping -c 4 8.8.8.8
$ ping -c 4 google.com

# Layer 4 — what's listening on this box?
$ sudo ss -tulpn

# Is a remote port open?
$ nc -zv example.com 443
$ nc -zv example.com 22 2>&1 | head -1

# DNS
$ dig +short example.com
$ dig +short MX example.com
$ getent hosts example.com

# HTTP
$ curl -I https://example.com
$ curl -v https://example.com 2>&1 | head -30
Mentor habit · start at L3

When an app can't reach a service, can you ping it? If not, the problem is routing/firewall — not config or auth. Many wasted hours come from debugging at L7 when the issue is at L3.

§5Gotchas

Gotcha · ifconfig is deprecated

Use ip. They're not 1:1 — ip -4 addr show eth0, not ifconfig eth0.

Gotcha · ping can be blocked

A box that doesn't ping may still serve HTTP fine. Don't conclude "down" from ping alone.

Mentor note · nc -zv proves port is accepting

Not that the service is alive. Use a protocol-aware test for L7.

Gotcha · DNS caching

System, browser, and upstream resolver all cache. sudo systemd-resolve --flush-caches or restart systemd-resolved.

Mentor note · ports below 1024

Require root to bind. Services drop privileges after binding.

Tip · ss -tulpn needs root for PIDs

Without sudo you see the sockets but not the owning process.

§6On-the-spot exercises

E18.1What IPv4 address(es) does this box have?

Show answer
$ ip -4 addr | grep inet
$ hostname -I

E18.2What's the default gateway?

Show answer
$ ip route | grep default

E18.3Is sshd listening? Which port?

Show answer
$ sudo ss -tulpn | grep ssh

E18.4Can you reach example.com:443?

Show answer
$ nc -zv example.com 443
$ curl -I https://example.com

E18.5What's example.com's IPv4?

Show answer
$ dig +short example.com

E18.6Trace route to 1.1.1.1, max 8 hops.

Show answer
$ traceroute -m 8 1.1.1.1
$ mtr -n 1.1.1.1

mtr is much friendlier — continuous output with packet-loss stats per hop.

E18.7Show the TLS certificate for example.com.

Show answer
$ echo | openssl s_client -connect example.com:443 -servername example.com 2>/dev/null \
  | openssl x509 -noout -subject -dates -issuer

E18.8Run a tiny HTTP listener on port 8000.

Show answer
# Terminal 1
$ python3 -m http.server 8000

# Terminal 2
$ curl -I http://localhost:8000/

Useful for quick file shares or testing reverse-proxy configs.

§7Real-world sysadmin scenario

"Web app can't reach the database." Classic layered debug:

# 1. Is the DB host name resolving?
$ dig +short db.internal

# 2. Can I reach the DB IP at all (L3)?
$ ping -c 2 db.internal

# 3. Is the DB port open (L4)?
$ nc -zv db.internal 5432

# 4. Is the DB *responding* on that port (L7)?
$ psql -h db.internal -U appuser -d appdb -c 'SELECT 1;'

Each step rules out one whole layer of bug. Compare to opening application logs and guessing — totally different speed of fix.

Another classic — "SSH suddenly stopped working":

$ ping target-host            # L3
$ nc -zv target-host 22       # L4
$ ssh -v target-host          # L7 — verbose ssh shows handshake details

Verbose ssh -v tells you "auth method publickey failed" vs "Connection refused" vs "Permission denied (publickey)" — three different problems with the same surface symptom.

§8What to remember

  • Debug bottom-up: link → IP → port → protocol.
  • ip replaces ifconfig; ss replaces netstat.
  • ss -tulpn is your "what's listening" one-liner.
  • dig +short for DNS without ceremony.
  • curl -v for HTTP; openssl s_client for TLS.
  • ping succeeds ≠ service alive. nc -zv is L4; still need L7 test.
