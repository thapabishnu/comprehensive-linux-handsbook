Services and systemd

systemctl and journalctl are how you find out why something won't start.

§1Why this matters

On modern Linux, systemd is PID 1 — the parent of everything. It manages services, mounts, sockets, timers, and the boot sequence.

§2Mental model

systemd organizes everything into units. A unit is a small declarative file saying: "here's what I want running, here's how to start it."

SuffixWhat it manages
.serviceA long-running process
.timerSchedules another unit (modern cron)
.socketA listening socket that activates a service on connect
.targetA group of units (like a runlevel)
.mountA filesystem mount
PathPurpose
/lib/systemd/system/Vendor-shipped unit files
/etc/systemd/system/Local admin overrides (highest priority)

A unit can be enabled (start at boot) and/or active (running right now). They're independent.

§3Core commands

CommandEffect
systemctl status nginxWhat's nginx's state right now?
systemctl start nginxStart now
systemctl stop nginxStop
systemctl restart nginxStop + start
systemctl reload nginxReload config (if unit supports it)
systemctl enable nginxStart automatically at boot
systemctl disable nginxDon't start at boot
systemctl enable --now nginxEnable and start in one step
systemctl is-active nginxJust "active" or "inactive"
systemctl --failedAnything in failed state
systemctl daemon-reloadReload systemd's own knowledge after unit edits
systemctl cat nginxShow unit file (vendor + overrides)
systemctl edit nginxCreate/edit override drop-in

Logs side

CommandEffect
journalctl -u nginxAll log entries for nginx
journalctl -u nginx -fFollow (like tail -f)
journalctl -u nginx --since "10 min ago"Time-bounded
journalctl -u nginx -bJust from this boot
journalctl -p errOnly error-priority and above
journalctl -kKernel messages
journalctl -xeLast entries with explanation, useful after a failure

§4Anatomy of a unit file

[Unit]
Description=My Web App
After=network.target

[Service]
Type=simple
User=myapp
WorkingDirectory=/srv/myapp
ExecStart=/srv/myapp/run.sh
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
  • [Unit] — description and ordering (After=, Before=, Requires=).
  • [Service] — how to run it.
  • [Install] — what enable should link it to.

Type=simple is most common. Type=notify is cleanest for production apps that can tell systemd "I'm ready."

§5Guided walkthrough

$ systemctl status ssh
$ systemctl is-active ssh
$ systemctl is-enabled ssh
$ systemctl list-units --type=service | head -10
$ systemctl --failed

# logs
$ journalctl -u ssh --since "1 hour ago"
$ journalctl -u ssh -f &
$ sudo systemctl restart ssh
$ fg
# Ctrl+C to stop tail

# create your own service
$ sudo tee /etc/systemd/system/hello.service > /dev/null <<'EOF'
[Unit]
Description=Hello world service
After=network.target

[Service]
Type=simple
ExecStart=/bin/bash -c 'while true; do echo "hello $(date)"; sleep 30; done'
Restart=on-failure

[Install]
WantedBy=multi-user.target
EOF

$ sudo systemctl daemon-reload
$ sudo systemctl enable --now hello
$ systemctl status hello
$ journalctl -u hello -f

# cleanup
$ sudo systemctl disable --now hello
$ sudo rm /etc/systemd/system/hello.service
$ sudo systemctl daemon-reload
Mentor note · daemon-reload required

After any change to a unit file. Forgetting this is the source of "I edited the unit but the change didn't take effect."

Drop-in overrides

Don't edit a vendor unit directly — your changes will be overwritten on package upgrade. Use systemctl edit:

$ sudo systemctl edit nginx

This creates /etc/systemd/system/nginx.service.d/override.conf — a drop-in that adds to specific directives:

[Service]
LimitNOFILE=65536

Then daemon-reload and restart.

§6Gotchas

Gotcha · reload needs ExecReload=

If the unit doesn't declare ExecReload, reload silently does nothing.

Gotcha · restart kills in-flight connections

Reload is gentler when supported.

Mentor note · journal persistence

journalctl shows history only if /var/log/journal/ exists. Set Storage=persistent in /etc/systemd/journald.conf and mkdir /var/log/journal.

Mentor note · failed units stay failed

systemctl reset-failed nginx clears the failed state.

Gotcha · After= is not Requires=

After=foo means "start me after foo if foo is started." Use Requires=foo (hard) or Wants=foo (soft) for actual dependencies.

§7On-the-spot exercises

E17.1Is the SSH service running?

Show answer
$ systemctl is-active ssh
$ systemctl status ssh

E17.2List every service that's failed.

Show answer
$ systemctl --failed
$ systemctl list-units --state=failed

E17.3Restart cron and confirm it's running.

Show answer
$ sudo systemctl restart cron
$ systemctl status cron

E17.4Disable bluetooth from starting at boot but leave it running.

Show answer
$ sudo systemctl disable bluetooth
$ systemctl is-enabled bluetooth   # → disabled
$ systemctl is-active bluetooth    # → still active

Use disable --now to also stop immediately.

E17.5Show SSH log entries from the last 10 minutes.

Show answer
$ journalctl -u ssh --since "10 min ago"

E17.6Tail-follow nginx's journal.

Show answer
$ journalctl -u nginx -f

E17.7Reload nginx config without restarting.

Show answer
$ sudo nginx -t                     # always validate first
$ sudo systemctl reload nginx

Skipping nginx -t has caused outages. Don't.

E17.8Override nginx to increase file-descriptor limit to 65536.

Show answer
$ sudo systemctl edit nginx
[Service]
LimitNOFILE=65536
$ sudo systemctl daemon-reload
$ sudo systemctl restart nginx
$ cat /proc/$(pgrep -f 'nginx: master' | head -1)/limits | grep "Max open files"

§8Real-world sysadmin scenario

"The website is down."

# 1. Is the service running?
$ systemctl status nginx
# Active: failed (Result: exit-code) since Fri 2026-05-16 14:32:01 UTC

# 2. Why did it fail?
$ journalctl -u nginx --since "5 min ago" -p err | tail -20
# nginx: [emerg] unknown directive "wokrer_processes" in /etc/nginx/nginx.conf:1

# 3. Fix the typo
$ sudo vi /etc/nginx/nginx.conf

# 4. Validate before reload
$ sudo nginx -t
# nginx: configuration file /etc/nginx/nginx.conf test is successful

# 5. Start it back up
$ sudo systemctl start nginx
$ systemctl is-active nginx
# active

Always systemctl status and journalctl before you start randomly restarting things.

§9What to remember

  • systemd is PID 1. Units describe what should run.
  • systemctl status / start / stop / restart / reload / enable / disable.
  • journalctl -u UNIT for logs; -f to follow.
  • After unit-file changes: systemctl daemon-reload.
  • Use systemctl edit for overrides; never edit vendor units directly.
  • reload over restart when supported. Validate configs first.
