MODULE19 MAPS TOLABS 62–64 TIME~35 min

SSH keys and 2FA

Keys solve guessability; 2FA solves the "key got stolen" problem. The modern minimum.

§1Why this matters

Passwords on SSH are weak — guessable, phishable, shared across machines. Keys solve guessability; 2FA solves the "key got stolen" problem.

§2Mental model

Public-key authentication: you have a private key (a long secret on your laptop) and a public key (safe to share). The server stores your public key in ~/.ssh/authorized_keys. The private key never leaves your laptop.

2FA layers a time-based one-time password (TOTP). After your key proves "what you have," TOTP proves "what you also have right now."

        ┌── factor 1: SSH key on laptop        ──┐
        │                                        ├──► access
        └── factor 2: TOTP code from phone (30s) ─┘

§3Core commands

Keys

CommandEffect
ssh-keygen -t ed25519 -C "you@laptop"Generate a modern keypair
ssh-keygen -t rsa -b 4096 -C "you@laptop"Older but universal
ssh-keygen -pChange passphrase on an existing key
ssh-keygen -l -f ~/.ssh/id_ed25519Show key fingerprint
ssh-copy-id user@hostInstall your public key on the server
cat ~/.ssh/id_ed25519.pubPrint your public key (safe to share)

Agent

CommandEffect
eval "$(ssh-agent -s)"Start the agent
ssh-add ~/.ssh/id_ed25519Load the key
ssh-add -lList loaded keys
ssh-add -DDrop all keys

SSH client config

~/.ssh/config lets you save per-host aliases:

Host prod-web
  HostName web-01.example.com
  User deploy
  Port 22
  IdentityFile ~/.ssh/id_ed25519
  ForwardAgent no

Host *
  ServerAliveInterval 60
  ServerAliveCountMax 3

Then ssh prod-web is enough.

§4Guided walkthrough — passwordless login

# 1. Generate a key
$ ssh-keygen -t ed25519 -C "you@$(hostname)"
# accept default path; choose a passphrase

# 2. View the public key
$ cat ~/.ssh/id_ed25519.pub

# 3. Install on the remote
$ ssh-copy-id you@remote-host

# 4. Verify
$ ssh you@remote-host
# should not prompt for password

# 5. Permissions sanity check (on remote)
$ ls -ld ~/.ssh ~/.ssh/authorized_keys
# expect:
# drwx------  ~/.ssh
# -rw-------  ~/.ssh/authorized_keys

If permissions are wrong, OpenSSH silently refuses to use the key.

Loading into ssh-agent

$ eval "$(ssh-agent -s)"
$ ssh-add ~/.ssh/id_ed25519     # asks passphrase
$ ssh-add -l                     # see it loaded
$ ssh you@remote                 # no passphrase prompt now

§5Adding Google Authenticator 2FA

Debian/Ubuntu:

$ sudo apt install -y libpam-google-authenticator
$ google-authenticator
# answer: y to time-based, y to update file, y to disallow reuse
# scan QR with Google Authenticator app; SAVE the emergency codes

Edit PAM config:

$ sudo vi /etc/pam.d/sshd

Add at the top:

auth required pam_google_authenticator.so

Edit SSH server config:

$ sudo vi /etc/ssh/sshd_config

Set:

ChallengeResponseAuthentication yes
UsePAM yes
AuthenticationMethods publickey,keyboard-interactive

That requires both the SSH key AND the TOTP code.

Mentor caution · open a second session first

Open a second SSH session before restarting sshd — if you've miswritten the config, the first session stays alive while you fix it. A locked-out box is a very long Saturday.

$ sudo systemctl restart ssh

§6Gotchas

Gotcha · permissions matter

~/.ssh = 700, authorized_keys = 600, private key = 600. Or OpenSSH refuses.

Gotcha · never share the private key

Share id_ed25519.pub. The one without .pub stays on your laptop.

Mentor note · passphrase your key

"But I'd rather not type it" — that's what ssh-agent is for. An unpassphrased key is a single point of failure.

Tip · disable password auth after keys work

Set PasswordAuthentication no in /etc/ssh/sshd_config. Test before logging out.

Gotcha · time skew breaks 2FA

TOTP codes are 30-second windows. If the server's clock drifts, codes fail. timedatectl status should show "NTP synchronized: yes."

§7On-the-spot exercises

E19.1Generate a new SSH key called ~/.ssh/id_test_ed25519.

Show answer
$ ssh-keygen -t ed25519 -f ~/.ssh/id_test_ed25519 -C "test"

E19.2Show the fingerprint of your default key.

Show answer
$ ssh-keygen -l -f ~/.ssh/id_ed25519

E19.3Print the public portion of your default key.

Show answer
$ cat ~/.ssh/id_ed25519.pub

E19.4Install your public key on localhost.

Show answer
$ ssh-copy-id "$USER@localhost"
$ ssh "$USER@localhost" whoami

E19.5Load your key into the agent, list, then remove it.

Show answer
$ eval "$(ssh-agent -s)"
$ ssh-add ~/.ssh/id_ed25519
$ ssh-add -l
$ ssh-add -d ~/.ssh/id_ed25519

E19.6Write a ~/.ssh/config entry so ssh box works with custom port/user/key.

Show answer
$ mkdir -p ~/.ssh && chmod 700 ~/.ssh
$ cat >> ~/.ssh/config <<'EOF'
Host box
  HostName 192.168.1.50
  Port 2222
  User admin
  IdentityFile ~/.ssh/id_box
EOF
$ chmod 600 ~/.ssh/config

E19.7Disable password authentication on a server (after confirming keys work).

Show answer
# keep a second session open
$ sudo sed -i.bak 's/^#\?PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
$ sudo sshd -t
$ sudo systemctl reload ssh

Show answer

Follow §5 step-by-step. Test in a new shell while keeping the original logged in.

§8Real-world sysadmin scenario

You inherited a fleet of 30 servers that still use password SSH:

# 1. Generate one strong key
$ ssh-keygen -t ed25519 -C "you@$(hostname)"

# 2. Get the public key onto every box
$ for host in $(cat hosts.txt); do
>   ssh-copy-id you@"$host"
> done

# 3. Verify
$ for host in $(cat hosts.txt); do
>   ssh -o BatchMode=yes -o StrictHostKeyChecking=no you@"$host" hostname 2>/dev/null \
>     || echo "FAIL: $host"
> done

# 4. Once all green, disable password SSH
$ for host in $(cat hosts.txt); do
>   ssh you@"$host" "
>     sudo sed -i.bak 's/^#\?PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config &&
>     sudo sshd -t &&
>     sudo systemctl reload ssh
>   "
> done

For the very sensitive servers, layer 2FA on top.

§9What to remember

  • Public key on the server; private key on your laptop only.
  • Permissions: ~/.ssh = 700, files = 600. Or OpenSSH refuses.
  • Passphrase your key. Use ssh-agent.
  • ~/.ssh/config saves you from typing hostnames and flags.
  • 2FA via Google Authenticator + PAM + AuthenticationMethods publickey,keyboard-interactive.
  • Always keep a second SSH session open when changing sshd config.
